Three structured documents were authored before any code generation began.
These constrained the AI's output and ensured architectural consistency.

3.1 Engineering Guidelines (ai-blueprint/engineering-guidelines.md)
    Purpose  : Define non-negotiable coding standards the AI must follow.
    Key Rules:
      - TypeScript strict mode everywhere; no plain .js except config files
      - Layered backend: routes → controllers → services → db pool
      - No ORM; raw parameterized SQL (mysql2/promise) only
      - JWT: 15-min access token in memory, 7-day refresh in httpOnly cookie
      - Error handling via typed AppError + centralized errorHandler middleware
      - Standard API envelope: { success, data, meta }
      - Conventional commits; .env gitignored, .env.example committed

3.2 Capability Definitions (ai-blueprint/capability-definitions.md)
    Purpose  : Describe the 9 functional building blocks the AI could compose.
    Modules Defined:
      1. Authentication   — JWT pair, refresh rotation, AuthContext, RequireAuth
      2. Product Catalog  — paginated list, filters, search, detail view
      3. Cart             — persistent server-side cart, CartContext, CartDrawer
      4. Checkout         — multi-step flow, address form, mock payment, confirmation
      5. Order Management — order history list + detail view with status badges
      6. User Profile     — editable name, change password
      7. UI Primitives    — Button, Badge, Spinner, Skeleton, Modal, Pagination
      8. Data Access      — pool.ts pattern, query pattern, transaction pattern
      9. Environment Cfg  — config.ts typed constants, never direct process.env

3.3 Bootstrap Prompt (ai-blueprint/initial.md)
    Purpose  : Primary instruction set to initiate full project generation.
    Contents :
      - Full folder structure specification for frontend and backend
      - Complete MySQL DDL requirements (8 tables with constraints)
      - Seed data requirements: 20+ products with Unsplash images, 4 categories
      - All API route definitions with method, path, auth, request/response shape
      - Frontend page list with routing and lazy-loading requirements
      - Docker Compose specification for db + backend + frontend services

6. PROMPT LOG — DEBUGGING & REFINEMENT PHASE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

D-01 | Type Mismatch: IAddress & IOrderItem
  Issue  : Frontend used street/postalCode; backend used fullName/line1/zip.
           IOrderItem used name/imageUrl; DB columns were productName/productImageUrl.
  Fix    : Updated both backend and frontend types/index.ts to align. Updated
           SQL queries to alias columns (productName AS name). Updated Zod schema
           in orderController to match new IAddress shape.

D-02 | MySQL caching_sha2_password Blocking Local Connections
  Issue  : Backend (running on host) could not authenticate against MySQL 8.0
           in Docker. Error: ER_ACCESS_DENIED_ERROR for lumina@172.18.0.1.
           MySQL 8 defaults to caching_sha2_password which requires SSL or
           RSA key exchange not supported by the mysql2 driver in this context.
  Fix    : ALTER USER 'lumina'@'%' IDENTIFIED WITH mysql_native_password BY ...
  Query  : "Why does mysql2 get access denied with MySQL 8.0 Docker container
           even with correct credentials and wildcard host?"

D-03 | Schema DB Name Mismatch
  Issue  : schema.sql used CREATE DATABASE lumina_store; compose created lumina.
  Fix    : Replaced all lumina_store references with lumina in schema.sql and seed.sql.
           Applied schema manually via PowerShell pipe: Get-Content ... | docker exec -i

D-04 | imageUrl Field Removal
  Issue  : Products and cart items had a redundant imageUrl field alongside the
           images JSON array. Caused type inconsistency and unnecessary DB column.
  Fix    : Removed imageUrl from schema, seed, types (frontend + backend), services,
           and components. Updated all usages to images[0] for thumbnail display.

D-05 | LIMIT Clause with mysql2 Parameterized Queries
  Issue  : mysql2 does not support LIMIT/OFFSET as parameterized ? placeholders
           when using pool.execute() — values were being cast to strings.
  Fix    : Interpolated safeLimit and offset directly into the SQL string after
           clamping/validating them as integers (Math.floor + bounds check).

D-06 | Register Error Message Swallowed
  Issue  : RegisterPage catch block showed hardcoded string for all failure types,
           hiding the real backend error (e.g., network error vs. 409 conflict).
  Fix    : Changed catch(err: unknown) to extract err.response.data.message with
           fallback. Applied same fix to LoginPage.tsx.

D-07 | Auth Redirect Loop on Register/Login Pages
  Issue  : Authenticated users could navigate to /login and /register, causing
           confusing UX.
  Fix    : Added RedirectIfAuth wrapper component in App.tsx — redirects to
           /products if already authenticated.

D-08 | Axios Interceptor Redirect on Auth Pages
  Issue  : Token refresh failure in client.ts triggered window.location.href='/login'
           even when already on /login or /register, causing reload loops.
  Fix    : Added pathname check before redirect in the refresh error handler.

8. HUMAN INTERVENTIONS (AI-GAP ANALYSIS)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The following required direct human judgment, not prompt tuning:

1. MySQL Auth Plugin Decision
   The AI could not diagnose or resolve the caching_sha2_password issue
   autonomously — it required knowing that mysql2 + Docker MySQL 8 requires
   mysql_native_password for non-SSL host connections. This is a runtime
   environment-specific issue outside the AI's training scope for this config.

2. PowerShell Pipe Syntax for Schema Import
   The AI attempted bash-style `<` redirect which PowerShell doesn't support.
   Human intervention was needed to use Get-Content ... | docker exec -i syntax.

3. Schema/Seed DB Name Correction
   The AI-generated schema used lumina_store but Docker Compose created lumina.
   Human identified the mismatch by reading both files and corrected all references.

4. imageUrl Architectural Decision
   The decision to remove the redundant imageUrl column (keeping only images[])
   was a product design decision. The AI executed the refactor once directed.

5. Vite Proxy Target for Docker
   Changing the Vite proxy target from localhost:4000 to backend:4000 for
   Docker networking was a human-initiated config change requiring Docker
   networking knowledge.

9. KEY ARCHITECTURAL DECISIONS MADE DURING INTERACTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Decision                        Rationale
──────────────────────────────  ───────────────────────────────────────────────
No ORM (raw mysql2 SQL)         Assignment requirement; full control over query
                                shape; no abstraction leakage into types.

Access token in memory only     Prevents XSS token theft; refresh via httpOnly
                                cookie is the industry standard pattern.

Single cart per user (server)   Simplifies sync; avoids client-side persistence
                                issues across devices.

Free shipping threshold $100    Business rule baked into orderService; easy to
                                surface in UI without config overhead.

mysql_native_password plugin    Required for mysql2 driver compatibility with
                                MySQL 8.0 in Docker without SSL.

Inline LIMIT in SQL string      mysql2 parameterized queries cast LIMIT values
                                to strings; inline integers (post-validation)
                                is the correct workaround.

images[] replaces imageUrl      Single source of truth for product images;
                                first element used as thumbnail everywhere.

orderStatus utility module      Centralized status → badge variant mapping to
                                avoid duplicated logic across 2+ pages.
